DOCUMENTACION TECNICA EXTENDIDA - SudoAgile
===========================================

Version del documento: 1.0
Objetivo: documentacion de continuidad para equipos de desarrollo y para alimentar agentes IA de soporte tecnico/funcional.
Alcance: arquitectura, dominio, seguridad, endpoints, flujos, convenciones, operacion local, riesgos y lineamientos de evolucion.


1) VISION TECNICA DEL SISTEMA
-----------------------------
SudoAgile es una aplicacion full stack para gestion agil de trabajo, con jerarquia central:

Proyecto -> Modulo -> Tarea -> Subtarea

Capacidades principales:
- autenticacion JWT con control de acceso por roles,
- CRUD + operaciones de flujo para entidades de gestion,
- asociacion de usuarios y prioridades por nivel jerarquico,
- gestion de adjuntos por tipo (documento, producto, marco, etc.),
- archivado y eliminacion logica para proyectos,
- exportacion de proyectos a Excel.

6) SEGURIDAD Y AUTENTICACION
----------------------------
Archivo clave:
- application/.../config/SecurityConfiguration.java

Puntos relevantes:
- seguridad stateless (SessionCreationPolicy.STATELESS),
- filtro JWT antes de UsernamePasswordAuthenticationFilter,
- /api/v1/autenticacion/** permitido sin autenticacion,
- autorizacion por hasAnyAuthority en rutas de negocio,
- /api/prioridad/** aparece permitido,
- CORS habilitado via CorsConfigurationSource.

CORS:
- application/.../config/CorsConfig.java
- origenes permitidos por defecto:
  - http://localhost:5173
  - http://127.0.0.1:5173

Autenticacion REST:
- /api/v1/autenticacion/signin
- /api/v1/autenticacion/signupuser
- /api/v1/autenticacion/signupadmin
- /api/v1/autenticacion/validateToken

Convencion frontend:
- token JWT en localStorage ("token")
- axios interceptor agrega Authorization: Bearer <token>

--------------------------------

2. crear modulos,
3. crear tareas,
4. crear subtareas,
5. asignar usuarios y prioridad,

13) CONVENCIONES DE CODIGO Y EVOLUCION
--------------------------------------
Backend:
- mantener flujo controller -> service -> repository,
- separar DTOs de entidades cuando aplique,
- evitar logica de negocio en controlador,
- estandarizar respuestas de error.

Frontend:
- concentrar llamadas HTTP en services/,
- minimizar llamadas directas en componentes,
- usar hooks para logica reusable,
- mantener naming consistente (es/es-EN) para reducir deuda.

Cross-cutting:
- evitar duplicacion de endpoints o variantes ambiguas,
- centralizar manejo de errores de red/auth.


14) OBSERVACIONES TECNICAS PARA CONTINUIDAD
-------------------------------------------
Hallazgos relevantes:
- coexistencia de estilos de consumo HTTP (instance y axios directo),
- rutas con nomenclaturas heterogeneas (camelCase/mixtas),
- algunos controladores mezclan mensajes y estructuras de respuesta distintas.

Recomendaciones de mejora incremental:
1. estandarizar contrato de error (code, message, details),
2. unificar cliente HTTP frontend,
3. documentar OpenAPI para estabilizar integraciones,
4. alinear estrategia de perfiles local/cloud,
5. normalizar convenciones de nombres en rutas y payloads.

16) RUNBOOK DE INCIDENTES
-------------------------
Incidente: 401/403 recurrente
Pasos:
1. validar token en frontend,
2. revisar expiracion del JWT,
3. confirmar autoridad del rol en SecurityConfiguration,
4. revisar que endpoint sea el esperado.

Incidente: CORS en desarrollo
Pasos:
1. validar origen frontend (5173),
2. revisar CorsConfig,
3. confirmar uso de URL/proxy correcto.

Incidente: datos no visibles por perfil
Pasos:
1. validar roles del usuario con /api/usuarios/getCurrentUser,
2. validar filtros por gestor/area funcional,
3. revisar endpoint usado en frontend.

Incidente: falla en adjuntos
Pasos:
1. confirmar endpoint de tipo correcto (documento/producto/etc.),
2. validar envio multipart/form-data,
3. revisar permisos y payload.
